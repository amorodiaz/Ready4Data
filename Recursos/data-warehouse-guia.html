1
El problema: cuando los datos están por todas partes

Imagina una empresa mediana. Tiene ventas en Shopify, clientes en Salesforce, contabilidad en SAP, soporte en Zendesk y métricas de marketing en Google Analytics.

El director comercial quiere saber: "¿Cuánto hemos vendido este mes a clientes que vinieron por Google Ads y que también han abierto un ticket de soporte?"

Imposible responder sin juntar datos de 4 sistemas distintos. Cada uno tiene su propio formato, su propia forma de identificar a los clientes, sus propios tiempos de actualización.

🏠 La analogía del armario

Imagina que tu ropa está repartida entre el armario del dormitorio, el trastero, la lavandería y casa de tu madre. Cada vez que quieres vestirte, tienes que buscarlo todo. Un Data Warehouse es el armario central donde está todo organizado.

Eso es exactamente el problema que resuelve un Data Warehouse (DW): concentrar todos los datos de la empresa en un único lugar, limpio, organizado y optimizado para responder preguntas de negocio.

2
La solución: qué es exactamente un Data Warehouse

Un Data Warehouse es una base de datos especial diseñada para análisis. A diferencia de las bases de datos operacionales (que están optimizadas para transacciones rápidas como registrar un pedido), un DW está optimizado para leer grandes cantidades de datos históricos y responder preguntas complejas.

Definición simple: Un Data Warehouse es el almacén centralizado donde se guardan y organizan todos los datos históricos de una empresa, listos para ser analizados.

Características clave:

  • Orientado al tema: organizado por áreas de negocio (ventas, RRHH, finanzas), no por aplicación
  • Integrado: datos de múltiples fuentes unificados con las mismas definiciones
  • No volátil: los datos no se borran ni modifican, solo se añaden
  • Variable en el tiempo: guarda historia. Puedes ver cómo era un cliente hace 2 años.

3
Cómo llegan los datos: el proceso ETL

Los datos no aparecen solos en el Data Warehouse. Hay un proceso llamado ETL (Extract, Transform, Load) que los mueve desde los sistemas fuente.

  • Extract (Extraer): sacar los datos de las fuentes (Salesforce, MySQL, archivos CSV, APIs…)
  • Transform (Transformar): limpiarlos, unificarlos, darles el formato correcto
  • Load (Cargar): meterlos en el Data Warehouse
🏭 La analogía de la fábrica

Imagina una fábrica de zumos. La materia prima (naranjas de distintos proveedores) llega sucia y en distintos tamaños. Se lava, se procesa y se embotella de forma estándar. El producto final (zumo en botella) es el Data Warehouse: limpio, estandarizado, listo para consumir.

En las arquitecturas modernas se habla más de ELT: primero cargas los datos crudos (Extract + Load) y luego los transformas dentro del propio Data Warehouse. Esto es posible gracias al bajo coste del almacenamiento en cloud y herramientas como dbt.

4
Cómo se organiza: el modelo dimensional

Los datos en un DW no se organizan igual que en una base de datos transaccional. El modelo más común es el esquema estrella, que tiene:

  • Tabla de hechos (fact table): contiene las métricas del negocio. Ejemplo: tabla de ventas con columnas importe, cantidad, fecha_id, cliente_id, producto_id.
  • Tablas de dimensiones (dim tables): describen el contexto. Ejemplo: dim_cliente (nombre, ciudad, segmento), dim_producto (nombre, categoría, precio), dim_tiempo (fecha, mes, año, trimestre).

Regla de oro: La tabla de hechos tiene los números. Las dimensiones tienen el contexto. Para responder "¿cuánto (hecho) vendimos en Madrid (dimensión) en enero (dimensión)?", haces un JOIN entre las tres.

Este diseño permite responder cualquier pregunta de negocio combinando la tabla de hechos con las dimensiones que necesitas.

5
El ecosistema moderno: Data Warehouse en la nube

Los Data Warehouses tradicionales (Oracle, Teradata) eran caros y complejos. La revolución cloud cambió todo. Hoy las opciones más usadas son:

HerramientaProveedorCaracterística
BigQueryGoogle CloudServerless, pago por consulta. El más usado en startups.
SnowflakeMulti-cloudSepara cómputo y almacenamiento. Muy escalable.
RedshiftAmazon AWSIntegrado con el ecosistema AWS.
Azure SynapseMicrosoftIntegrado con Power BI y el mundo Microsoft.
DatabricksMulti-cloudLakehouse: combina DW y Data Lake.

El stack moderno más común para empresas medianas es: Fuentes → Fivetran/Airbyte (extracción) → Snowflake/BigQuery (almacén) → dbt (transformación) → Power BI/Tableau (visualización).

Para recordar: El Data Warehouse no es el destino final, es el puente entre los datos crudos y las decisiones de negocio. Su valor se mide en preguntas que antes no podías responder y ahora sí.
